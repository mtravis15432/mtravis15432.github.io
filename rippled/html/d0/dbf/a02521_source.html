1 ## Bootstrap Strategy
2 
3 Fresh endpoints are ones we have seen recently via mtENDPOINTS.
4 These are best to give out to someone who needs additional
5 connections as quickly as possible, since it is very likely
6 that the fresh endpoints have open incoming slots.
7 
8 Reliable endpoints are ones which are highly likely to be
9 connectible over long periods of time. They might not necessarily
10 have an incoming slot, but they are good for bootstrapping when
11 there are no peers yet. Typically these are what we would want
12 to store in a database or local config file for a future launch.

29 The process of obtaining the initial peer connections for accessing the Ripple
30 peer to peer network, when there are no current connections, is called
31 "bootstrapping." The algorithm is as follows:
32 
33 1. If ( unusedLiveEndpoints.count() > 0
34  OR activeConnectionAttempts.count() > 0)
35  Try addresses from unusedLiveEndpoints
36  return;
37 2. If ( domainNames.count() > 0 AND (
38  unusedBootstrapIPs.count() == 0
39  OR activeNameResolutions.count() > 0) )
40  ForOneOrMore (DomainName that hasn't been resolved recently)
41  Contact DomainName and add entries to the unusedBootstrapIPs
42  return;
43 3. If (unusedBootstrapIPs.count() > 0)
44  Try addresses from unusedBootstrapIPs
45  return;
46 4. Try entries from [ips]
47 5. Try entries from [ips_urls]
48 6. Increment generation number and go to 1

52 "Connection Strategy"
53 ---------------------
54 
55 This is the overall strategy a peer uses to maintain its position in the Ripple
56 network graph
57 
58 We define these values:
59 
60  peerCount (calculated)
61  The number of currently connected and established peers
62 
63  outCount (calculated)
64  The number of peers in PeerCount that are outbound connections.
65 
66  MinOutCount (hard-coded constant)
67  The minimum number of OutCount we want. This also puts a floor
68  on PeerCount. This protects against sybil attacks and makes
69  sure that ledgers can get retrieved reliably.
70  10 is the proposed value.
71 
72  MaxPeerCount (a constant set in the rippled.cfg)
73  The maximum number of peer connections, inbound or outbound,
74  that a peer wishes to maintain. Setting MaxPeerCount equal to
75  or below MinOutCount would disallow incoming connections.
76 
77  OutPercent (a baked-in program constant for now)
78  The peer's target value for OutCount. When the value of OutCount
79  is below this number, the peer will employ the Outgoing Strategy
80  to raise its value of OutCount. This value is initially a constant
81  in the program, defined by the developers. However, it
82  may be changed through the consensus process.
83  15% is a proposed value.
84 
85 However, lets consider the case where OutDesired is exactly equal to MaxPeerCount / 2.
86 In this case, a stable state will be reached when every peer is full, and
87 has exactly the same number of inbound and outbound connections. The problem
88 here is that there are now no available incoming connection slots. No new
89 peers can enter the network.
90 
91 Lets consider the case where OutDesired is exactly equal to (MaxPeerCount / 2) - 1.
92 The stable state for this network (assuming all peers can accept incoming) will
93 leave us with network degree equal to MaxPeerCount - 2, with all peers having two
94 available incoming connection slots. The global number of incoming connection slots
95 will be equal to twice the number of nodes on the network. While this might seem to
96 be a desirable outcome, note that the connectedness (degree of the overlay) plays
97 a large part in determining the levels of traffic and ability to receive validations
98 from desired nodes. Having every node with available incoming connections also
99 means that entries in pong caches will continually fall out with new values and
100 information will become less useful.
101 
102 For this reason, we advise that the value of OutDesired be fractional. Upon startup,
103 a node will use its node ID (its 160 bit unique ID) to decide whether to round the
104 value of OutDesired up or down. Using this method, we can precisely control the
105 global number of available incoming connection slots.

107 "Outgoing Strategy"
108 -------------------
109 
110 This is the method a peer uses to establish outgoing connections into the
111 Ripple network.
112 
113 A peer whose PeerCount is zero will use these steps:
114  1. Attempt addresses from a local database of addresses
115  2. Attempt addresses from a set of "well known" domains in rippled.cfg
116 
117 
118 This is the method used by a peer that is already connected to the Ripple network,
